# 会话架构：多会话 vs 单会话

## 问题背景

用户可能在不同 tab、不同设备、不同时间发起多轮对话。每轮对话对应一个 LangGraph **thread**。如何设计，使得：
- 计划、进度、饮食等数据在用户视角是「连贯的」
- 不同会话之间能共享信息

---

## 方案对比

### 方案 A：多会话共享（当前采用）

**含义**：同一用户可同时存在多个 thread（会话）。每个 thread 对应一次「新对话」。但用户数据（计划、进度、饮食等）存放在 **共享存储**：`workspace/<user_id>/`。

| 维度 | 说明 |
|------|------|
| 数据存储 | `workspace/<user_id>/`，按 user_id 隔离，跨 thread 共享 |
| 计划 | `workout/plans/current.json`，唯一 |
| 进度 | `workout/session_state.json`，唯一 |
| 新对话 | 新 thread，但读同一 workspace |

**优点**：
- 符合多端使用：手机、电脑、多 tab 都能看到同一数据
- 实现简单：workspace 已按 user_id 设计
- 用户可随时「开新对话」问问题，不打断原有对话

**缺点**：
- 多会话同时写时可能有时序问题（可加简单锁或「最后写入胜出」）
- 对话历史分散在多个 thread，无法在一个界面看到「和教练的全部对话」

---

### 方案 B：单会话（每用户一个 thread）

**含义**：每个用户只有一个「主 thread」。新对话都复用该 thread，或强制「继续上次对话」。

| 维度 | 说明 |
|------|------|
| 数据存储 | 同上，但 thread 与 user 一一对应 |
| 实现 | 需维护 user_id → thread_id 映射；新对话时复用该 thread |

**优点**：
- 符合「和教练一对一」的心智：同一段对话延续
- 对话历史完整，便于回顾

**缺点**：
- 实现复杂：需在 DB 或配置中存 user_id → thread_id
- 多端同时打开时，可能「抢」同一 thread
- 用户想「开新话题」时，需要显式「新对话」功能

---

## 当前决策：方案 A（多会话共享）

**理由**：
1. 架构已支持：workspace 按 user_id，跨 thread 共享
2. 多端、多 tab 更自然
3. 实现成本低：只需确保 agent 读/写正确路径，并主动记录

**实现要点**：
- 计划、进度、饮食等一律在 `workspace/<user_id>/` 下
- Agent 每次涉及这些数据时，**先读**已有文件，再决定创建/修改
- 用户提到训练进度时，**主动**更新 `session_state.json`

---

## 若未来改为方案 B

需要：
1. 用户表或配置中增加 `primary_thread_id`
2. 登录/创建 thread 时，若用户已有 primary_thread_id，则复用
3. 前端「新对话」时，可选择「继续上次」或「真正新建」（新建时更新 primary_thread_id）

当前阶段建议保持方案 A，待产品验证后再考虑方案 B。
